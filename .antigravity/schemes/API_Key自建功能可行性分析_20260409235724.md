### 2.5 ⚠️ 前端 Feature Flag — 方案遗漏的关键点

> [!CAUTION]
> **方案遗漏了一个关键障碍**：前端 API Key 菜单入口受 `Feature.API_KEYS`（值为 `'api:keys'`）控制。

前端通过 `POST /workspace/entitlements` 获取 feature 列表：

```typescript
// license-check.service.ts → resolveFeatures()
// 无 EE 模块时返回空数组 []
// → 前端 hasFeature('api:keys') 为 false
// → API Key 菜单项被禁用（显示但灰色不可点击）
```

**因此，除了方案中提到的后端改动外，还需要修改 `resolveFeatures()` 或 `entitlements` 端点让其返回 `'api:keys'` feature。**

## 四、与方案对比的补充和修正

### 4.1 ⚠️ 修正：不应修改 `app.module.ts`

方案建议在 `app.module.ts` 注册 `ApiKeyModule`，但更好的做法是在 `core.module.ts` 中注册，因为其他业务模块（User, Auth, Page 等 ）都在 `core.module.ts`。

### 4.2 ⚠️ 新增：Feature Flag 解锁

方案未提到需要处理前端 entitlements。有两种方式：
1. **修改 `resolveFeatures()`**：在无 license 时也返回 `['api:keys']`（最小改动）
2. **修改 `entitlements` 端点**：硬编码返回 `api:keys`

### 4.3 ✅ 确认：JWT 作为 API Key

方案中提到的"生成 JWT 作为 API Key"思路是对的。从 `jwt.strategy.ts` 的 `validate()` 逻辑看，Bearer token 就是一个标准 JWT，解码后通过 `type` 字段分发到不同的验证逻辑。

### 4.4 ✅ 确认：前端无需额外改动

前端组件、服务、路由全部就绪。调用的 API 端点（`/api-keys`、`/api-keys/create`、`/api-keys/update`、`/api-keys/revoke`）只需后端实现即可。
