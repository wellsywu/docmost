# API_Key自建功能执行计划

## 1. 选择的执行方案简介
依据前期的可行性分析报告，本计划将在 Docmost 社区版中自建 API Key 认证功能：
- **解锁界面入口**：修改 Feature Flag 控制点 `license-check.service.ts`，使前端正常开放配置入口。
- **构建核心逻辑**：基于现有 JWT 生态，新增 API Key CRUD（`core/api-key`）及哈希存储能力。
- **替换验证节点**：修改企业版 JWT Token 的验证拦截器 `jwt.strategy.ts`，由自建的 `ApiKeyService` 接管 `validateApiKey()` 过程。

本方案在数据库存储 `key_hash`，不存储明文 token，完全符合已有 API Key 安全规范。

## 2. 执行步骤及效果预测

### 步骤 1：前端入口解锁
- **动作**：修改 `apps/server/src/integrations/environment/license-check.service.ts`，在 `resolveFeatures()` 返回的数组中默认追加 `'api:keys'`。
- **预测**：重启服务后，工作空间侧边栏和个人账号侧边栏会出现 "API keys" 与 "API management" 菜单项。

### 步骤 2：数据库改造
- **动作**：编写并执行迁移脚本 `apps/server/src/database/migrations/20260410T000000-api-key-hash.ts`，为 `api_keys` 表添加 `key_hash` (TEXT, UNIQUE) 字段。
- **预测**：数据库结构增加验证所需的哈希对比校验字段，满足后续模块操作。

### 步骤 3：接口模型与业务开发
- **动作**：在 `apps/server/src/core/api-key/` 目录下完成 `Module`, `Controller`, `Service`, `DTO` 等编写。
- **预测**：形成独立子模块支持创建、验证、列表、撤销 API Key，并成功衔接数据库 ORM。

### 步骤 4：认证总线切换
- **动作**：修改 `apps/server/src/core/auth/strategies/jwt.strategy.ts` 将 EE 模块验证切为本地 `ApiKeyService`，修改 `core.module.ts` 加入 `ApiKeyModule`。
- **预测**：前端面板完全打通，携带 `Authorization: Bearer <API Key>` 时可顺利通过所有接口校验，功能 100% 解锁。
