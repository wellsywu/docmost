# API Key 功能测试部署方案

> 制定时间：2026-04-10
> 适用范围：Docmost 社区版自建 API Key 功能测试及上线

---

## 1. 原理与背景

通过直接修改源码核心认证层与前端鉴权控制点，在不用打包 EE 模块的前提下，全流程打通 API Key 面板和底层认证接口。
因修改了 Kysely 迁移脚本及核心 Guard 控制逻辑，需进行严格测试，确保已有业务（页面交互、现有鉴权）不受影响。

## 2. 部署与测试执行步骤

### 2.1 本地测试前置准备
1. 确保已配置好本地需要的环境变量（如 `.env.local` 里的 `DATABASE_URL` , `REDIS_URL` 和 `APP_SECRET`）。
2. 安装完整依赖库（若是第一次）：
   ```bash
   pnpm install
   ```

### 2.2 运行数据库迁移 (关键)
本次修改涵盖了新增 `key_hash` 列的迁移文件 `20260410T000000-api-key-hash.ts`。需要执行以下命令同步表结构：
```bash
# 在 server 端或项目根目录执行对应 migration 命令
pnpm --filter=@docmost/server run migration:up
```

*验证点：* 使用任意数据库客端连接 DB，检查 `api_keys` 表末尾是否增加了 `key_hash` (TEXT, null permitted or unique)。

### 2.3 启动开发服务器测试
在项目根目录启动完整的 Turbo 代码流进行编译 + 热重载：
```bash
pnpm dev
```
或者指定只跑 client / server：
```bash
pnpm --filter=@docmost/server run dev
pnpm --filter=@docmost/client run dev
```

### 2.4 测试用例清单 (Test Cases)

#### 前端交互验证路线
1. **入口可见性**：
   - 登录系统，左下角进入『设置』(Settings)。
   - 验证 "Account" 层级下菜单是否包含 `API keys`。
   - 验证 "Workspace" 层级下面（若是 Owner 或者是 Admin 身份），是否包含 `API management` 菜单。
   - **预期**：均正常可见，不再处于 disabled 灰色且没有不可点 tooltip。

2. **创建 API Key**：
   - 进入 `API keys` 页面，点击 `Create API Key` 按钮。
   - 输入有效名称并保存。
   - **预期**：弹出模态框展示带有复制图标的绿色 Base64 JWT 字符串（这就是我们的 API Key）。关闭模态框后不会再看到明文。

3. **管理与撤销**：
   - 创建后的 Key 会出现在表格。
   - 尝试更改它的 Name，或者直接操作 Revoke 删除。
   - **预期**：成功保存名称或完全撤销。状态应及时刷新表单，且不会引发控制台 500 报错。

#### 后端能力验证路线 (CURL/Postman测试)
1. 拿到上述刚刚创建的 API Key。
2. 调用 `/api/workspace/info` 或等价安全路由：
   ```bash
   curl -X POST http://localhost:3000/api/workspace/info \
        -H "Authorization: Bearer <刚才复制出来的API_KEY>" \
        -H "Content-Type: application/json"
   ```
3. **预期成功结果**：返回 HTTP 200 及 `{"id": "...", "name": "...", "plan": "..."}`。
4. 取一条撤销过的 API Key 发起相同请求。
   - **预期失败结果**：返回 `401 Unauthorized` 和错误消息『API Key 不存在或已被撤销』。

### 2.5 生产部署指南
若本地测试完全通关，即可开始服务器部署：
1. **提交代码**：将基于 `master` 增加的代码 (`core/api-key`、迁移文件、`jwt.strategy.ts` 等) Commit。
2. **重新构建镜像**：
   ```bash
   docker build -t docmost-custom .
   ```
3. **滚动更新**：
   在服务器使用 Docker Compose 重新拉起容器（使用新镜像），在此过程 Docker Entrypoint 通常会自动完成新迁移挂载 `migration:up` 的执行。
   ```bash
   docker-compose up -d --build
   ```

## 3. 风险与回退 (Rollback)

如果本地跑完发生了不可逆的拦截错误或编译不过（通常不常见因为 `core.module.ts` 及业务文件是标准 NestJS）：
1. 倒退数据库（移除 `key_hash` 迁移）：
   ```bash
   pnpm --filter=@docmost/server run migration:down
   ```
2. 回退代码：`git checkout apps/server/src/integrations/environment/license-check.service.ts` 等发生变化的代码点。
